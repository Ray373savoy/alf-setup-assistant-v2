# 検証および質問ガイド

Task JSON 生成前の「検証および質問」段階で確認すべき事項と質問例です。

---

## チェックリスト

| 項目 | 確認内容 | 質問例 |
|------|----------|--------|
| 🔴 注文照合方法 | 注文の照合方法が会話内で確定していない場合、必ずユーザーに確認する | （下記「注文照合方法の確認」セクション参照） |
| 🔴 memorySchema完全性 | コードノード内の全 memory.put/get キーが memorySchema に登録されているか | — （質問不要、自動チェック） |
| 🔴 memorySchema型一致 | memory.put で保存する値の型と memorySchema の type が一致するか（配列→list、オブジェクト→object） | — （質問不要、自動チェック） |
| ノードタイプ | メッセージノードで変数を使用？→ agent に変更 | 「このノードは memory 変数を使用する必要があるため、エージェントノードが適切だと思います。変更しましょうか？」 |
| ノードタイプ | 動的会話なしの固定メッセージ？→ message に変更 | 「このノードは固定の案内のみなので、メッセージノードで十分だと思います。どうしましょうか？」 |
| 不要なノード | メッセージノード → エージェントノードの連続？ | 「このメッセージノードの案内をエージェントノードでまとめれば統合できそうです。統合しましょうか？」 |
| 情報ソース | 名前/メール/電話番号等、会員なら入力不要？ | 「会員の場合、電話番号を別途入力せずに context から取得しますか？」 |

### 🔴 注文照合方法の確認（ステップ2で必ず実施）

注文を確認するフローを設計する場合、**照合方法がクライアントによって異なる**。
タスク設計の会話内で照合方法が明確に指定されていない場合、以下の質問で確認すること。

**確認質問（そのまま使用可）：**

```
注文の照合方法について確認させてください。
お客様の注文を特定する際、以下のようなパターンがあります：

パターンA: 注文番号のみヒアリング → 会員IDと自動突合
  - チャネルトークの会員情報（context.user.memberId）と注文の顧客IDを照合
  - 会員ログイン済みの場合に有効

パターンB: 注文番号 + お名前 + メールアドレスをヒアリング → 3点照合
  - 注文に紐づく名前・メールアドレスと入力値を突合
  - 非会員や会員IDが連携されていないケースで使用

パターンC: 注文番号 + 電話番号をヒアリング → 2点照合

パターンD: その他（独自の照合方法）

どのパターンで照合しますか？
また、照合に失敗した場合（名前が一致しない等）の対応はオペレーター接続でよいですか？
```

**照合パターンごとの設計への影響：**

| パターン | ヒアリング項目 | agentノード | 照合ロジック（codeノード） | memorySchema 追加キー |
|---------|-------------|------------|------------------------|---------------------|
| A: 会員ID自動突合 | 注文番号のみ | 1ノード（番号ヒアリング） | `context.user.memberId` と注文の `customer_id` を比較 | なし（contextから取得） |
| B: 3点照合 | 注文番号・名前・メール | 1ノード（3点まとめてヒアリング）or 複数ノード | 入力値と注文情報の名前・メールを比較 | `input_customer_name`, `input_customer_email` |
| C: 2点照合 | 注文番号・電話番号 | 1ノード（2点ヒアリング） | 入力値と注文情報の電話番号を比較 | `input_customer_phone` |
| D: 独自 | クライアント指定 | クライアント要件に従う | クライアント要件に従う | クライアント要件に従う |

⚠️ 照合に失敗した場合（identity_match = false）のフローも確認すること。
通常は「ご本人確認が取れませんでした。担当者にお繋ぎします」→ オペレーター接続。

### 🔴 memorySchema 完全性チェック手順（JSON出力前に必ず実施）

1. 全コードノードのコードから `memory.put('xxx', ...)` のキーを抽出する
2. 全コードノードのコードから `memory.get('xxx')` のキーを抽出する
3. 全エージェントノードの instruction から `<promptdata ... identifier="xxx">` のキーを抽出する
4. 全ブランチ条件から `taskMemory.xxx` のキーを抽出する
5. 上記1〜4の全キーが `memorySchema` に存在するか確認する
6. **1件でも不足があれば memorySchema に追加してから JSON を出力する**

```
チェック例:
  コードノード node-2: memory.put('order_search_status', ...) → memorySchema に order_search_status あり ✅
  コードノード node-2: memory.put('shopify_order_id', ...)    → memorySchema に shopify_order_id あり ✅
  コードノード node-2: memory.put('errorMessage', ...)        → memorySchema に errorMessage なし ❌ → 追加必須
  ブランチ node-3: taskMemory.order_search_status              → memorySchema に order_search_status あり ✅
```

## 例1: ノードタイプ変更の提案

### 状況
FigJam で「配送状況案内」が**メッセージノード**として描かれている。
しかし案内内容に `配送状況`、`到着予定日` などの変数が含まれている。

### 質問
```
🔍 「配送状況案内」ノードを確認しました。

このノードで配送状況、到着予定日などの変数を使用する必要がありそうです。
メッセージノードでは memory 変数を活用できないため、**エージェントノード**がより適切だと思います。

エージェントノードに変更しましょうか？
```

### 回答例
- 「はい、変更してください」→ エージェントノードに設計変更
- 「いいえ、固定メッセージです」→ メッセージノードを維持

## 例2: 不要なノードの検出

### 状況
FigJam で「案内メッセージ1」→「案内メッセージ2」の2つの連続したメッセージノードがある。
両ノードとも単純なテキスト案内で、間に分岐やロジックがない。

### 質問
```
🔍 連続したメッセージノードが2つあります。

「案内メッセージ1」→「案内メッセージ2」

2つのノード間に分岐や顧客応答の待機がなければ、
1つのメッセージノードに統合できそうです。

統合しましょうか、それとも別々に維持する理由がありますか？
```

### 回答例
- 「統合してください」→ 1つのメッセージノードに統合
- 「間にディレイが必要です」→ 別々に維持

## 例3: 分岐の完全性確認

### 状況
FigJam で「決済手段確認」分岐がある。
- カード → A ノード
- ネイバーペイ → B ノード
- （default なし）

### 質問
```
🔍 「決済手段確認」分岐を確認しました。

現在の分岐:
- カード → カード返金案内
- ネイバーペイ → ネイバーペイ返金案内

しかし **カカオペイ、口座振込、携帯決済** など他の決済手段はどう処理しますか？
1. 相談員接続
2. 一般的な返金案内（別途メッセージ）
3. 現在の2つのみ対応（残りはなし）

どの方式にしましょうか？
```

### 回答例
- 「1番です、相談員接続で」→ default を相談員接続に設定
- 「3番、カードとネイバーペイだけです」→ default を END_TASK に

## 例4: エージェントノードの終了条件

### 状況
FigJam で「顧客情報収集」エージェントノードがある。
終了条件が明示されていない。

### 質問
```
🔍 「顧客情報収集」エージェントノードの終了条件が明確ではありません。

このノードはいつ終了すべきですか？
1. 顧客が情報を入力したら（例: 名前、連絡先）
2. 案内メッセージ出力後すぐに
3. 顧客が「完了」「次へ」などと応答したら

どの条件にしましょうか？
```

### 回答例
- 「1番、名前と連絡先を受け取ったらです」→ memory に保存時の終了条件を設定
- 「2番、案内だけすればいいです」→ 案内出力後の終了条件を設定

## 例5: 同一ノードの共有 vs 分離

### 状況
FigJam で複数の分岐から「相談員接続」ノードに合流している。
- 分岐 A（y=200）→ 相談員接続
- 分岐 B（y=1500）→ 相談員接続（同一ノード）

### 質問
```
🔍 複数の分岐で「相談員接続」ノードを共有しています。

現在の構造:
- 分岐 A（上側）→ 相談員接続
- 分岐 B（下側）→ 相談員接続（同一ノード）

2つの分岐の Y 座標の差が大きいと、エディタで線が絡まる可能性があります。
**別々のノードに分離**しましょうか、それとも**共有を維持**しましょうか？

（タグや案内メッセージが異なる場合は分離を推奨）
```

### 回答例
- 「分離してください」→ 各分岐に別々の相談員接続ノードを作成
- 「共有で大丈夫です」→ 1つのノードを維持

## 質問の原則

1. **質問総数をまず案内**: 「全部で N 個の質問があります。1番目の質問です。」
2. **1つずつ質問**: 質問を1つずつ順番に進行
3. **具体的な選択肢を提示**: 「どうしましょうか？」より「1、2、3のどれにしますか？」
4. **理由を説明**: なぜこの質問をするのか背景を説明
5. **提案を含める**: 単純な質問よりも「〜が適切だと思います」の形を推奨
