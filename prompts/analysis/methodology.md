# Task 複雑度測定方法論

## 概要

Channel.io Taskシステムの複雑度を客観的に測定するための多次元分析フレームワークです。

## 複雑度測定の次元

### 1. ワークフロー複雑度 (Workflow Complexity)

#### 1.1 ノード構造複雑度
- **コアノード数**：コードノードとエージェントノードの個数（メッセージ、相談処理アクションノード除外）
- **最大実行パス長**：開始から終了まで最も長いパスのノード数
- **分岐点数**：Branchノードの個数

```javascript
// 計算公式（修正版）
ノード_構造_スコア = (
  (コア_ノード数 / 8) * 0.4 +
  (最大_パス長 / 12) * 0.35 +
  (分岐点_数 / 6) * 0.25
) * 10

// コアノード：code、agentノードのみ計算
// 除外ノード：message、userChatInlineActionノード
```

#### 1.2 分岐複雑度
- **条件分岐数**：各Branchノードの条件個数
- **ネスト分岐深度**：分岐内分岐の最大深度
- **循環複雑度**：McCabe循環複雑度を適用

### 2. 技術的複雑度 (Technical Complexity)

#### 2.1 コード複雑度
- **API呼び出し複雑度**：外部サービス連携数とAPI呼び出し数
- **データ処理複雑度**：JSONパース、変換、計算ロジックの複雑性
- **エラー処理**：try-catchブロックと例外処理ロジック（小規模スコア）

```javascript
// 技術的複雑度計算公式（修正版）
技術的_複雑度_スコア = (
  (外部_サービス_連携_数 / 4) * 0.35 +
  (API_呼び出し_数 / 8) * 0.25 +
  (データ_処理_複雑度 / 5) * 0.2 +
  (メモリ_変数_個数 / 10) * 0.15 +
  (エラー_処理_スコア / 3) * 0.05
) * 10

// 外部サービス例：Cafe24、Slack、EzAdmin、Booster Labs等
// データ処理：JSONパース、日付計算、データマッピング等
// メモリ変数：Task実行中に使用される変数
// エラー処理：try-catch、状態検証、例外状況処理
```

#### 2.2 メモリ管理複雑度
- **メモリ変数個数**：Taskで使用されるメモリ変数の総個数

### 3. ビジネスロジック複雑度 (Business Logic Complexity)

#### 3.1 ドメイン複雑度
- **ビジネスルール数**：実装されたビジネスルールの個数
- **条件付きロジック**：複雑なif-else条件文
- **状態管理**：多様な状態と状態遷移ロジック

```javascript
// ビジネスロジック複雑度計算公式
ビジネス_ロジック_複雑度_スコア = (
  (agent_ノード_instruction_長さ_合計 / 1000文字) * 0.4 +
  (分岐_条件_filter_複雑度 / 10) * 0.35 +
  (agent_ノード_steps_個数 / 20) * 0.25
) * 10

// agent_ノード_instruction_長さ_合計：すべてのagentノードのinstructionテキスト長の合計
// 分岐_条件_filter_複雑度：すべてのbranchノードのfilter条件複雑度の合計
// agent_ノード_steps_個数：すべてのagentノードのsteps配列項目数の合計
```

#### 3.2 Agentノード複雑度測定基準
- **Instruction長**：instruction/stepsテキストの総長さ（長い指示文＝複雑なロジック）
- **Promptdataタグ数**：`<promptdata>`タグ使用個数（メモリ変数参照頻度）
- **対話フロー段階**：instruction内のナンバリングされた段階数
- **終了条件複雑度**：終了条件の個数と複雑性

#### 3.3 分岐条件複雑度測定基準
- **ネスト深度**：`filter.and/or`構造のネスト深度
- **条件個数**：key-operator-valuesの組み合わせの総個数
- **演算子重み付け**：
  - 単純演算子（$eq、$exist）：1点
  - 比較演算子（$gt、$lt、$gte、$lte）：2点
  - 配列演算子（$containsAny、$ncontainsAny）：2点
  - 複雑演算子（$regex、$in）：3点

```javascript
// 分岐条件複雑度計算例
分岐_条件_複雑度 = (
  条件_個数 * 平均_演算子_重み +
  ネスト_深度 * 2
)
```

## 複雑度スコア計算

### 全体複雑度公式

```javascript
総_複雑度_スコア = (
  ワークフロー_複雑度 * 0.35 +
  技術的_複雑度 * 0.40 +
  ビジネス_ロジック_複雑度 * 0.25
)

// スコア範囲：1-10（1：非常に単純、10：非常に複雑）
```

### 複雑度等級分類

| スコア範囲 | 等級 | 説明 | 予想開発時間 | 予想価格 |
|-----------|------|------|----------------|------------|
| 1.0 - 2.5 | 単純 | 基本的な線形フロー | 1-2日 | 20-25万円 |
| 2.6 - 4.0 | 普通 | いくつかの分岐とAPI呼び出し | 3-5日 | 25-30万円 |
| 4.1 - 6.0 | 複雑 | 多重分岐と複雑なロジック | 1-2週間 | 30-37万円 |
| 6.1 - 8.0 | 非常に複雑 | 高度なビジネスロジック | 2-3週間 | 37-43万円 |
| 8.1 - 10.0 | 極めて複雑 | 多重システム連携 | 3-4週間 | 43-50万円 |

## 価格設定

### 価格算定公式

複雑度スコアに基づいて開発コストを算定します：

```javascript
// 線形マッピング：複雑度1点＝20万円、10点＝50万円
予想_価格(万円) = 20 + (総_複雑度_スコア - 1) * (30 / 9)

// または簡単に
予想_価格(万円) = 20 + (総_複雑度_スコア - 1) * 3.333
```

### 価格算定根拠

- **最低価格（20万円）**：単純な線形Taskの基本開発コスト
- **最高価格（50万円）**：極めて複雑な多重システム連携Task
- **線形増加**：複雑度増加に伴う開発時間とコストの比例関係

### 価格調整要素

実際のプロジェクトでは以下の要素を追加で考慮できます：

- **緊急度**：納期が逼迫している場合 +10-20%
- **保守運用**：長期保守運用契約時 +15-25%
- **カスタマイズ**：特殊要件反映時は個別算定
- **大量割引**：複数Task同時開発時 -5-10%
