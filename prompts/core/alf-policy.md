# ALF v2 ポリシードキュメント (ALF v2 Policy Documentation)

本ドキュメントは Notion「企画 SSOT」データベースを基に整理された ALF v2 の核心ポリシー、課金モデル、マイグレーションシナリオおよびシステム制約事項を記述します。

---

## 1. 概要および移行ポリシー (Overview & Migration)

### 1.1 移行タイムライン

- **10/30 以前**: すべてのチャネルに `AlfRunMode` を作成。
- **10/30 〜 11/27 (共存期間)**:
  - 新規チャネル: V2 (Beta) をデフォルト適用。
  - 既存チャネル: V1 を維持、Backdoor を通じて V2 (Beta) への切り替えが可能。
  - 課金: V2 使用量は On-demand（別途 `frontAlfBeta` フィーチャー）で処理。
- **11/27 以降 (Official Release)**:
  - すべてのチャネルに **V2 を強制適用**。
  - クライアントで `AlfRunMode` を無視し V2 として動作。
  - V1 Add-on は V2 へマイグレーション（金額置換）。

### 1.2 運用モード (FrontAlfRunMode)

- **構成要素**:
  - `chatV2` (bool): チャットボット実行バージョン (Default: true)
  - `voiceV2` (bool): 音声ボット実行バージョン (Default: false)
  - `backdoor` (bool): V1/V2 切り替え UI の表示有無 (Readonly)
- **動作ロジック**:
  - `backdoor = true`: 設定 UI に V1/V2 切り替えオプションを表示。
  - `backdoor = false`: 設定 UI に V2 のみ表示。

## 2. 課金および使用量ポリシー (Billing & Usage)

### 2.1 コスト構造

- **ALF セッション**: セッションあたり **500 AU**（500ウォン相当）
- **Task 実行**: 実行あたり **200 AU**（200ウォン相当）
- **無料ポリシー**: 失敗した応答（ユーザーに有意な応答が到達しなかった場合）は課金対象外。

### 2.2 課金トリガー（ALF セッション生成）

ALF セッションは「有意な応答」が発生した際に生成および課金されます。

- **有意な応答の定義**:
  - 一般メッセージ応答 / RAG（ナレッジ）による応答。
  - Task 開始メッセージ。
  - ユーザーチャットアクション（相談オープン、タグ付与、変数アップデートなど）。
  - ユーザー言語の変更。

### 2.3 セッション終了条件

- マネージャー（相談員）が介入した場合（相談オープン）。
- ALF の最終応答から **24時間** が経過した場合。
- すでにオープンされたユーザーチャットで再度オープン動作を実行した場合。

### 2.4 プラン別提供量 (AU)

- **Trial**: 10,000 AU
- **Growth**: 30,000 AU（基本提供）
- **Startup**: 90,000 AU
- **Enterprise**: 100,000 AU
  _11/28 正式リリース時点から適用。_

## 3. 基本動作ルール (Basic Rules)

### 3.1 トーン＆マナー (Tone & Manner)

- **基調**: 専門的でありながら親しみやすく、わかりやすく明確な言語を使用。
- **スタイル**:
  - 回答は要点から頭括式で作成。
  - 長い内容はリスト（Bullet points）や段落で区分。
  - **分量制限**: 回答あたり **400文字以内**。
  - 文末に余白（改行）を処理。

### 3.2 応答精度および RAG 活用

- **RAG 優先**: 内部 FAQ やアーティクルの内容を最優先で活用。
- **変更禁止**: ポリシー性の文言、FAQ 回答は任意に修正・要約せず原文のまま引用。
- **出典提供**: リンクや参考資料がある場合は必ず併せて提供。
- **不明確な質問**: 推測的な回答を避け、具体的な情報を聞き返して確認。

### 3.3 共感および言語

- **共感表現**: 顧客の感情（当惑、不便など）を具体的に言及して共感した後、解決策を提示。形式的な謝罪は避ける。
- **言語サポート**: 顧客が使用する言語に合わせて回答（保存された言語設定より会話言語を優先）。

### 3.4 相談員接続 (Fallback)

- 顧客が強い不満・不便を示すか、相談員接続を明示的に要請した場合は即座に接続。

## 4. システム制約事項および Validation (Limits)

### 4.1 ルール (Rules)

- **数量制限**: チャネルあたり最大 **100個**（Active 状態は **30個** まで）。
- **文字数制限**: タイトル 50文字、指示事項（Instruction）2,000文字。
- **制約**: 稼働中（Active）のルールは削除不可。

### 4.2 ナレッジ (Knowledge)

- **数量制限**: チャネルあたりフォルダ含め合計 **100個**（要検討）。
- **構造制限**: フォルダ深度最大 **5階層**。
- **ファイル制限**: アップロードファイルサイズ **10MB** 以下。
- **文字数制限**: タイトル 50文字、内容 500文字。
- **制約**: フォルダ内にナレッジがあるか、参照中のナレッジは削除不可。

### 4.3 タスク (Tasks)

- **数量制限**: チャネルあたり最大 **100個**（Active 状態は **50個** まで）。
- **構造制限**: フォルダ深度 **1階層**。
- **制約**:
  - 稼働中のタスクは削除不可。
  - 循環（Cycle）構造は不可。
  - 未接続ノード（Isolated Node）は許容（保存可能、パブリッシュ不可）。

### 4.4 ノード別制約 (Node Validation)

- **AgentNode**: Instruction 10,000文字制限。
- **CodeNode**: 公式仕様上は制限なしだが、実運用では5,000文字超でインポートエラーの報告あり。5,000文字以内を目標とすること。
- **Task Memory**: Key 個数 50個、重複 Key 不可。
